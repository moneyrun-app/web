## 다음 세션에서 해야 할 것

### 백엔드 API 연동 테스트
백엔드 코스 API가 배포되면:
1. **온보딩 플로우 E2E 테스트** — 5단계 전체 완주, 이어하기 동작 확인
2. **미션 완료** — POST /course/missions/:id/complete 동작 확인
3. **코스 선택/시작** — POST /course/:id/start, 마이북 생성 폴링
4. **v2 레거시 유저** — 로그인 후 코스 UI 미표시 확인

### UI/UX 개선 (디자인 피드백 후)
- 온보딩 각 스텝 디자인 세부 조정 (간격, 색상, 모션)
- 미션 완료 시 축하 애니메이션
- 코스 완료 화면 디자인
- 다크모드 세부 색상 확인

### 누락된 기능
- **코스 완료 페이지** — `/course/complete` (현재 없음, `POST /course/active/complete` 후 보여줄 화면)
- **코스 시작 시 마이북 생성 폴링** — `POST /course/:id/start` 후 generating 상태 처리
- **코스 내 퀴즈 카테고리 스코핑** — 백엔드에서 처리하므로 프론트 변경 불필요

### 확인 필요 사항 (백엔드)
- `/auth/kakao` 응답에 `onboardingVersion` 필드 추가 여부
- 코스 API 엔드포인트 전체 배포 상태
- v2 미완료 유저의 `onboardingVersion` 기본값 확인

## 아키텍처 결정 사항 (참고)

| 결정 | 이유 |
|---|---|
| `(onboarding)/` 별도 라우트 그룹 | 인증 필요 + 네비 불필요 → `(app)`과 `(public)` 어디에도 안 맞음 |
| `useAuthInit` 공통 훅 | JWT 교환 로직 중복 방지. `(app)`과 `(onboarding)` 모두 사용 |
| 온보딩 상태는 서버가 source of truth | 클라이언트 onboardingStore는 프리필 편의용. 항상 GET /status로 확인 |
| step4 generate는 멱등 | 진입 시 무조건 POST 해도 안전. 재진입 분기 불필요 |
| 코스 데이터는 TanStack Query 캐시만 사용 | Zustand 스토어 불필요. 서버 상태를 중복 저장하지 않음 |
| v2 호환은 activeCourse null 체크로 | onboardingVersion 대신 데이터 존재 여부로 게이팅 |
